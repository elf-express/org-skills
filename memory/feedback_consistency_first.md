寫 org-skills 內任何規範性內容時,**先給定本(baseline)+ 偏離條件**,而不是給選項清單讓 reader 自選。

**Why**:用戶在 2026-05-14 搭建 `elf-express/org-skills` 倉庫時直接說:
> 「我主要是要把規範弄一致,不然每一次都不一樣」

這是他建立 skill 倉庫的核心動機 — 不是為了給團隊更多選擇,而是為了**消除「每次寫法不同」的不一致性**。Skill 對他而言是「凍結團隊共識」的載體。

**How to apply**:
- SKILL.md 的核心配置(playwright.config、API 命名、Entity 欄位等)應該寫成「**這就是團隊定本,起新專案 copy 即可**」,而不是「以下是候選 A、B、C,請依場景選擇」。
- 若有變體必須說明,**明列偏離條件**(例如「純前端 UI 測試可開 `fullyParallel: true`,API 測試維持 false」),而不是用「視情況而定」這種模糊話術。
- 寫 spec 時記決策的「為什麼」與「排除的方案」,讓未來想改規範的人理解前提,而不是讓他覺得規範是任意選擇。
- 回應用戶時若有多個方案,給推薦預設值 + 替代方案理由,而不是讓他從零選。
